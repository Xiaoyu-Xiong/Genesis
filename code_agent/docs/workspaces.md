# Workspaces

`code_agent/workspaces/` will contain generated run workspaces.

Run workspaces can become large. Commit only curated examples or small fixtures intentionally.

## Standard Layout

```text
code_agent/workspaces/<run_id>/
  inputs/
    user_prompt.md
    repo_rules.md
    capabilities.md
  contracts/
    planner_output.json
    scene_brief.json
    scene_plan.json
    module_contracts.json
    asset_requests.json
  assets/
    asset_manifest.json
  src/
    scene.py
    body.py
    action.py
    rendering.py
    main.py
  logs/
    codex_planner.jsonl
    codex_scene.jsonl
    codex_body.jsonl
    codex_action.jsonl
    codex_rendering.jsonl
    codex_integrator.jsonl
    codex_review.jsonl
    execution.stdout
    execution.stderr
  artifacts/
    run_result.json
    event_log.json
    metrics.json
    render_stats.json
    render.mp4
    frames/
  reports/
    static_review.json
    execution_report.json
    legacy_execution_report.json
    critic_report.json
    patch_plan.json
  summary.json
```

The current rigid MVP writes `summary.json`, `metrics.json`, `event_log.json`, `run_result.json`, and, when enabled,
render artifacts under each case's `artifacts/` directory, plus suite-level `summary.json` at the run root after a
suite completes.

In `--generation-mode codex`, `src/rendering.py` is generated by the dedicated Rendering Worker. In
`--generation-mode fallback`, rendering support is still minimal and may be absent or no-op depending on the fallback
template.

Worker logs include both `codex_<role>.jsonl` and `codex_<role>.final.md`. For current writers, the final message is a
JSON object containing `source_code`; the source files under `src/` are written by the coordinator after parsing that
JSON.
